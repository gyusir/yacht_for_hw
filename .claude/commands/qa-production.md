프로덕션 사이트(https://yacht-ff0c8.web.app)의 UI/UX QA 테스트를 수행합니다.

## 실행 방법

- `/qa-production` → 전체 QA (아래 모든 Phase 수행)
- `/qa-production 다크모드 주사위 스킨` → 해당 키워드에 관련된 항목에 포커스하여 집중 QA
- `/qa-production 방금 수정한 로비 레이아웃 확인` → 자유 텍스트로 QA 범위 지정 가능

### 포커스 모드 동작 규칙
- 추가 텍스트가 있으면(`$ARGUMENTS`), 해당 내용과 관련된 화면/기능만 집중 테스트한다.
- 전체 Phase를 순서대로 수행하지 않고, 관련 Phase만 선택적으로 수행한다.
- 단, Phase 1(초기 접속 및 콘솔 체크)은 항상 수행한다.
- 결과 보고 형식은 동일하게 유지한다.

**사용자 지시: $ARGUMENTS**

## 테스트 계정

로그인이 필요한 QA에는 메모리에 저장된 테스트 계정을 사용합니다. (memory/reference_qa_test_account.md 참조)
- 로그인 절차:
  1. "Google로 로그인" 버튼을 클릭
  2. Google 로그인 팝업/리다이렉트가 뜨면 사용자에게 "패스키 인증을 완료해주세요"라고 안내
  3. `browser_wait_for`로 로비 화면이 나타날 때까지 대기 (최대 30초)
  4. 로그인 완료 확인 후 QA 계속 진행
- 게스트 모드 테스트도 반드시 포함 (로그인 QA와 별도로 수행)

## QA 절차

### Phase 1: 초기 접속 및 콘솔 체크
1. `browser_navigate`로 https://yacht-ff0c8.web.app 접속
2. `browser_console_messages` (level: error)로 에러 확인
3. `browser_take_screenshot`으로 초기 화면 캡처

### Phase 2: 로그인 화면 검증
1. 스크린샷 + 스냅샷으로 구조 확인
2. 다크모드 전환 후 스크린샷
3. 영어 전환 후 스크린샷
4. 한국어 + 라이트모드로 복원
5. 확인 항목:
   - 제목, 버전 표시 정상 여부
   - "게스트로 플레이" / "Google로 로그인" 버튼 존재 및 클릭 가능 여부
   - "플레이 방법" 버튼 동작 여부
   - 다크/라이트 모드 전환 즉시 반영 여부
   - 한국어/영어 전환 정상 여부

### Phase 3: 로비 화면 검증
1. "게스트로 플레이" 클릭하여 로비 진입
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 닉네임 표시 정상 여부 (12자 이내)
   - 방 코드 입력 필드 및 "참여" 버튼 (disabled 상태)
   - "방 만들기" / "랜덤 매치" / "봇 대전" 버튼 존재 및 클릭 가능
   - "로그인으로 돌아가기" / "피드백" 링크 동작
   - 다크모드 전환 시 즉시 반영

### Phase 4: 봇 대전 설정 화면 검증
1. "봇 대전" 클릭
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 게임 모드 선택 (야찌/요트) UI 및 선택 상태 표시
   - 난이도 선택 (일반/도박사/해일) UI 및 선택 상태 표시
   - "게임 시작" / "로비로 돌아가기" 버튼
   - 각 옵션 클릭 시 선택 상태 전환

### Phase 5: 게임 화면 검증
1. 야찌 모드 + 일반 난이도로 게임 시작
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 스코어카드 렌더링 (13개 카테고리 + 보너스 + 야찌 보너스)
   - 주사위 5개 표시
   - "굴리기!" 버튼 (0/3 표시)
   - 이모트 버튼 영역
   - 상단 메뉴 (빨리감기, 단축키, 규칙, 무승부, 기권)
   - 플레이어 이름 컬럼 헤더
   - 합계 행 표시
4. "굴리기!" 클릭하여 주사위 굴림 동작 확인
5. 굴림 후 스크린샷 — 주사위에 값이 표시되는지, 스코어카드에 프리뷰가 나오는지 확인
6. 콘솔 에러 재확인

### Phase 6: 모바일 반응형 검증
1. viewport를 375x812 (iPhone SE)로 변경
2. 게임 화면 full-page 스크린샷
3. 확인 항목:
   - 스코어카드가 화면에 맞게 축소되는지
   - 주사위 터치 영역이 충분한지
   - 메뉴 버튼 접근 가능한지
   - 텍스트 잘림/오버플로우 없는지
4. 로비 화면으로 이동 후 모바일 스크린샷
5. viewport를 1280x800으로 복원

### Phase 7: 접근성 검증
1. `browser_snapshot`으로 접근성 트리 확인
2. 확인 항목:
   - 모든 버튼에 접근 가능한 이름이 있는지
   - 이미지에 alt 텍스트가 있는지
   - heading 계층 구조가 올바른지 (h1 > h2)
   - 폼 요소에 레이블이 있는지
   - disabled 상태가 올바르게 표시되는지

## 주의사항
- 각 화면 전환 후 반드시 스크린샷을 남깁니다
- 콘솔 에러는 Phase 1과 Phase 5 이후 최소 2회 확인합니다
- 다크모드 테스트는 반드시 라이트모드로 복원 후 다음 단계로 진행합니다
- 모바일 테스트 후 반드시 데스크톱 뷰포트로 복원합니다
