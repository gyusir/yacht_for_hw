로컬 에뮬레이터 환경(http://localhost:5002)의 UI/UX QA 테스트를 수행합니다.

## 실행 방법

- `/qa-emulator` → 전체 QA (아래 모든 Phase 수행)
- `/qa-emulator 다크모드 주사위 스킨` → 해당 키워드에 관련된 항목에 포커스하여 집중 QA
- `/qa-emulator 방금 수정한 로비 레이아웃 확인` → 자유 텍스트로 QA 범위 지정 가능

### 포커스 모드 동작 규칙
- 추가 텍스트가 있으면(`$ARGUMENTS`), 해당 내용과 관련된 화면/기능만 집중 테스트한다.
- 전체 Phase를 순서대로 수행하지 않고, 관련 Phase만 선택적으로 수행한다.
- 단, Phase 0(에뮬레이터 확인)과 Phase 1(초기 접속 및 콘솔 체크)은 항상 수행한다.
- 결과 보고 형식은 동일하게 유지한다.

**사용자 지시: $ARGUMENTS**

## 테스트 계정

에뮬레이터에는 사전 설정된 테스트 계정 2개가 있습니다 (`emulator-data/auth_export`에서 로드):

| 계정 | 이메일 | displayName | 용도 |
|---|---|---|---|
| 일반 | testuser@test.com | testuser | 일반 기능 테스트 |
| 긴 이름 | testforlongusername@test.com | testforlongusername | 긴 닉네임 UI 테스트 (12자 초과) |

- 에뮬레이터에서는 비밀번호 없이 이메일만으로 로그인 가능
- 두 계정 모두 테스트에 활용 (닉네임 잘림, 레이아웃 확인)

## QA 절차

### Phase 0: 에뮬레이터 확인 및 시작
1. `lsof -i:5002`로 에뮬레이터가 이미 실행 중인지 확인
2. 실행 중이면 그대로 Phase 1로 진행
3. 실행 중이 아니면 `/localtest` 스킬을 호출하여 에뮬레이터를 시작한 뒤 Phase 1로 진행

### Phase 1: 초기 접속 및 콘솔 체크
1. `browser_navigate`로 http://localhost:5002 접속
2. `browser_console_messages` (level: error)로 에러 확인
3. `browser_take_screenshot`으로 초기 화면 캡처
4. 에뮬레이터 연결 확인 — 콘솔에 에뮬레이터 관련 로그가 있는지, 프로덕션 서버로 연결되지 않았는지 확인

### Phase 2: 로그인 화면 검증
1. 스크린샷 + 스냅샷으로 구조 확인
2. 다크모드 전환 후 스크린샷
3. 영어 전환 후 스크린샷
4. 한국어 + 라이트모드로 복원
5. 확인 항목:
   - 제목, 버전 표시 정상 여부
   - "게스트로 플레이" / "Google로 로그인" 버튼 존재 및 클릭 가능 여부
   - "플레이 방법" 버튼 동작 여부
   - 다크/라이트 모드 전환 즉시 반영 여부
   - 한국어/영어 전환 정상 여부

### Phase 3: 로비 화면 검증
1. "게스트로 플레이" 클릭하여 로비 진입
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 닉네임 표시 정상 여부 (12자 이내)
   - 방 코드 입력 필드 및 "참여" 버튼 (disabled 상태)
   - "방 만들기" / "랜덤 매치" / "봇 대전" 버튼 존재 및 클릭 가능
   - "로그인으로 돌아가기" / "피드백" 링크 동작
   - 다크모드 전환 시 즉시 반영

### Phase 4: 봇 대전 설정 화면 검증
1. "봇 대전" 클릭
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 게임 모드 선택 (야찌/요트) UI 및 선택 상태 표시
   - 난이도 선택 (일반/도박사/해일) UI 및 선택 상태 표시
   - "게임 시작" / "로비로 돌아가기" 버튼
   - 각 옵션 클릭 시 선택 상태 전환

### Phase 5: 게임 화면 검증
1. 야찌 모드 + 일반 난이도로 게임 시작
2. 스크린샷 + 스냅샷
3. 확인 항목:
   - 스코어카드 렌더링 (13개 카테고리 + 보너스 + 야찌 보너스)
   - 주사위 5개 표시
   - "굴리기!" 버튼 (0/3 표시)
   - 이모트 버튼 영역
   - 상단 메뉴 (빨리감기, 단축키, 규칙, 무승부, 기권)
   - 플레이어 이름 컬럼 헤더
   - 합계 행 표시
4. "굴리기!" 클릭하여 주사위 굴림 동작 확인
5. 굴림 후 스크린샷 — 주사위에 값이 표시되는지, 스코어카드에 프리뷰가 나오는지 확인
6. 콘솔 에러 재확인

### Phase 6: 모바일 반응형 검증
1. viewport를 375x812 (iPhone SE)로 변경
2. 게임 화면 full-page 스크린샷
3. 확인 항목:
   - 스코어카드가 화면에 맞게 축소되는지
   - 주사위 터치 영역이 충분한지
   - 메뉴 버튼 접근 가능한지
   - 텍스트 잘림/오버플로우 없는지
4. 로비 화면으로 이동 후 모바일 스크린샷
5. viewport를 1280x800으로 복원

### Phase 7: 접근성 검증
1. `browser_snapshot`으로 접근성 트리 확인
2. 확인 항목:
   - 모든 버튼에 접근 가능한 이름이 있는지
   - 이미지에 alt 텍스트가 있는지
   - heading 계층 구조가 올바른지 (h1 > h2)
   - 폼 요소에 레이블이 있는지
   - disabled 상태가 올바르게 표시되는지

### Phase 8: 긴 닉네임 계정 검증
1. 로그아웃 후 testforlongusername@test.com 계정으로 로그인
2. 로비 및 게임 화면에서 닉네임 표시 확인
3. 확인 항목:
   - 닉네임이 12자에서 잘리는지
   - 잘린 닉네임이 레이아웃을 깨뜨리지 않는지
   - 모바일에서도 정상 표시되는지

## 주의사항
- localhost이면 자동으로 emulator에 연결된다.
- 각 화면 전환 후 반드시 스크린샷을 남긴다
- 콘솔 에러는 Phase 1과 Phase 5 이후 최소 2회 확인한다
- 다크모드 테스트는 반드시 라이트모드로 복원 후 다음 단계로 진행한다
- 모바일 테스트 후 반드시 데스크톱 뷰포트로 복원한다
- 에뮬레이터에서는 App Check가 비활성화되므로 관련 에러는 무시한다

## 무시해도 되는 콘솔 에러
- `signInWithPassword 400 (Bad Request)` — 에뮬레이터 계정 선택 과정에서 발생하는 예상된 에러. 기능에 영향 없음.

## 브라우저 세션 관리
- 이전 세션의 브라우저가 남아 "Browser is already in use" 에러가 발생하면:
  1. `browser_close` 시도
  2. 그래도 안 되면 `pkill -f "mcp-chrome"` 실행 후 재접속

## 스코어카드 카테고리 선택 패턴
게임 화면에서 카테고리를 선택할 때 **2단계 확정 방식**을 사용한다:
1. 첫 번째 클릭: 해당 카테고리가 `pending` 상태로 선택됨 (파란 테두리)
2. 두 번째 클릭: 확정됨

**주의**: 첫 번째 클릭 후 DOM이 재렌더링되어 기존 ref가 무효화될 수 있다.
이 경우 `browser_evaluate`로 직접 클릭한다:
```js
() => {
  const tds = document.querySelectorAll('td');
  for (const td of tds) {
    if (td.classList.contains('pending')) { td.click(); return 'confirmed'; }
  }
  return 'not found';
}
```

## 봇 턴 대기 시간
- 봇은 최대 3회 굴림 + stagger 애니메이션 + 카테고리 선택까지 수행한다
- 일반/도박사 봇: `wait_for` 5~6초
- 해일(Wave) 봇: `wait_for` 6~8초 (Endgame Worker 계산 포함 시 더 길어질 수 있음)
- 봇 턴 완료 확인: "굴리기! (0/3)" 텍스트가 나타나면 내 턴이 돌아온 것
